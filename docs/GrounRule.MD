# 💡 Team Ground Rule

## 1. 일정 및 협업 도구
- 주요 공동 작업 도구: **GitHub, Notion, Discord** 
- 정기 팀 미팅: 매주 토요일 오후 8시 디스코드에서 비대면으로 진행
ㄴ 필요시 유동적으로 추가 회의 진행 
- 부득이한 사정으로 회의에 불참할 경우 **최소 하루 전 사전 공유**
- 프로젝트 일정, 발표 일정, 마감일 및 주요 개인 일정은 **Notion 캘린더**에 기록하여 팀 전체가 공유
- 개인 담당 업무는 정해진 기한 내 완료를 원칙으로 하며, 지연이 예상될 경우 즉시 사유와 함께 공유하고 수정 일정을 제안
- 반복적인 미이행이 발생할 경우 벌금을 부여


## 2. 공동 작업 진행 방식

### 2.1 개발 작업 (GitHub)
- 모든 개발 작업은 GitHub을 기준으로 관리
- 각 팀원은 본인이 수행할 작업을 **Issue로 등록**하고 내용을 명시
- 작업 완료 후 **Pull Request**를 통해 코드 리뷰를 거친 뒤 병합
- 진행 상황과 변경 사항은 Issue 및 PR 기록을 통해 투명하게 공유

### 2.2 문서 및 기획 작업 (Notion)
- 기획 및 문서 작업은 Notion에서 관리
- 작업 항목, 담당자, 마감 기한을 명시하여 등록
- 회의록, 결정 사항, 참고 자료는 Notion에 정리하여 팀원 전체가 동일한 정보를 확인 가능하게 함

### 2.3 커뮤니케이션 (Kakaotalk)
- 일상적인 소통과 빠른 논의는 Kakaotalk을 통해 진행한다.
- 주요 안건은 공지로 등록 


## 3. Team Culture

> 📌 **팀 기조: 서로 존중하며 신뢰를 기반으로 협력하는 팀 운영**

### 3.3 의사소통

- 불편 사항이나 개선점은 감정이 아닌 내용 중심으로 솔직하게 공유
- 피드백은 **“상황 + 제안”** 방식으로 전달
- 질문이나 요청에는 **24시간 이내 응답**을 원칙
  - 즉시 답변이 어려운 경우 확인 메시지를 먼저 전달

### 3.2 시간 관리

- 회의 시작 **5분 전 입장**을 원칙으로 한다.
- 지각 또는 일정 변경이 필요한 경우 사전에 공유한다.
- 불참 시 회의록을 반드시 확인하고 후속 작업에 반영한다.


### 3.3 업무 수행 원칙

- 모든 결정 사항은 회의록 및 Notion 문서로 기록한다.
- 업무 진행 상황, 이슈, 어려움은 지연 없이 팀에 공유한다.
- 문제 발생 시 개인 책임보다는 **해결 방안 중심으로 논의**한다.
- 발표 또는 중요 일정에 불참하게 될 경우, 담당 부분의 PPT 및 대본을 사전에 준비하여 팀원에게 전달한다.



## 4. Code Convention

### 4.0 개요
- 팀에서 합의한 Code Style 지키기
- 주석 및 커밋 메시지를 명확하게 작성하기
- Pull Request 전 코드 리뷰 & 테스트 수행하기


### 4.1 이름 규칙 (Naming Rules)

| 구분        | Python              | JavaScript & Java      | 기타         |
|-------------|---------------------|------------------------|--------------|
| PascalCase  | 클래스, Exception   | 클래스, Exception      | -            |
| camelCase   | -                   | 변수, 함수, 메소드     | -            |
| snake_case  | 변수, 함수, 메소드  | -                      | -            |
| kebab-case  | -                   | -                      | 파일, 폴더명 |
| UPPER_CASE  | 상수변수            | 상수변수               | -            |

- **Boolean 타입 변수 규칙**
    1. **상태** → `isRunning`, `isEnabled`
    2. **소유** → `hasData`, `hasChildNodes`
    3. **가능** → `canSubmit`, `canAccess`
    4. **권장/조건** → `shouldUpdate`, `shouldDisplay`


### 4.2 주석 및 커밋

#### 📝 주석 작성 원칙
- 불필요한 주석(코드 그대로 반복하는 설명)은 작성하지 않는다.
- 변경 시 주석도 **반드시 최신 상태**로 유지한다.

#### 💬 커밋 메시지 작성 원칙
- 의미 단위별로 작은 단위로 커밋한다.
- 커밋 메시지는 무엇을, 왜 변경했는지를 명확히 담는다.
- **Prefix 규칙**
    - `feat:` 새로운 기능 추가
    - `fix:` 버그 수정
    - `docs:` 문서 수정
    - `refactor:` 기능 변경 없는 코드 구조 개선
    - `test:` 테스트 코드 추가/수정
    - `style:` UI 컴포넌트 변경 


### 4.3 Pull Request

#### PR 생성 전 체크리스트
- 로컬에서 정상적으로 실행 확인
- 주요 기능 테스트 실행
- 불필요한 디버깅 코드와 주석 제거
- 코드 스타일 준수

#### PR 작성 원칙
- 변경된 주요 내용 요약 작성
- 필요 시 테스트 스크린샷/실행 로그 첨부
- Reviewer가 빠르게 이해할 수 있도록 간단한 설명 제공

#### 리뷰 시 유의사항
- 코드 품질 우선 확인
- 단순 비난이 아닌 개선 제안 중심 피드백
- Merge 전 최소 1명 이상의 리뷰 필요
