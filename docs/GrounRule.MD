# 💡 Team Ground Rule

## 1. 일정 및 협업 도구
- 팀의 공식 협업 도구인 GitHub, Notion, Discord, Kakaotalk)을 통해 모든 작업과 소통에 참여
- 정기 팀 미팅(매주 토요일 오후 8시, Discord 비대면)에 온라인으로 참석
- 추가 회의가 잡힐 경우 가능한 일정은 우선 참여하고, 불참 시 최소 하루 전에 사유를 공유
- 회의에 참석하지 못한 경우 회의록을 확인하고, 확인 완료 후 피드백과 작업 진행 상황을 공유
- 프로젝트 일정, 마감일, 개인 일정은 Notion 캘린더에 직접 기록하고 수시로 확인


## 2. 공동 작업 진행 방식

### 2.1 개발 작업 (GitHub)
- 모든 개발 작업은 GitHub을 기준으로 관리
- 맡은 작업은 시작 전에 Issue로 등록
- 작업 진행 중간 결과도 커밋으로 기록
- 작업 완료 후 Pull Request를 생성하고 코드 리뷰를 받은 뒤 수정 및 반영
- Issue와 PR 기록으로 진행 상황과 변경 사항을 공유

### 2.2 문서 및 기획 작업 (Notion)
- 기획 및 문서 작업은 Notion에서 작성 및 수정
- 담당 작업, 마감 기한, 진행 상태를 직접 업데이트
- 회의록과 결정 사항을 확인하고 필요한 내용은 추가로 정리

### 2.3 커뮤니케이션 (Kakaotalk)
- 일상적인 소통과 빠른 논의는 Kakaotalk을 통해 진행
- 주요 안건은 공지로 등록 

### 2.4 작업 책임 기준
- 담당 작업은 회의 중 정한 기한 내 완료를 기준으로 함 
- 지연이 예상될 경우 즉시 사유와 수정 일정을 공유
- 팀에서 정한 규칙과 패널티 기준을 따름 


## 3. Team Culture

> 📌 **팀 기조: 서로 존중하며 신뢰를 기반으로 협력하는 팀 운영**

### 3.3 의사소통

- 불편 사항이나 개선점은 감정이 아닌 내용 중심으로 솔직하게 공유
- 피드백은 **“상황 + 제안”** 방식으로 전달
- 질문이나 요청에는 **24시간 이내 응답**을 원칙
  - 즉시 답변이 어려운 경우 확인 메시지를 먼저 전달

### 3.2 시간 관리

- 회의 시작 **5분 전 입장**을 원칙으로 한다.
- 지각 또는 일정 변경이 필요한 경우 사전에 공유한다.
- 불참 시 회의록을 반드시 확인하고 후속 작업에 반영한다.


### 3.3 업무 수행 원칙

- 모든 결정 사항은 회의록 및 Notion 문서로 기록한다.
- 업무 진행 상황, 이슈, 어려움은 지연 없이 팀에 공유한다.
- 문제 발생 시 개인 책임보다는 **해결 방안 중심으로 논의**한다.
- 발표 또는 중요 일정에 불참하게 될 경우, 담당 부분의 PPT 및 대본을 사전에 준비하여 팀원에게 전달한다.



## 4. Code Convention

### 4.0 개요
- 팀에서 합의한 Code Style 지키기
- 주석 및 커밋 메시지를 명확하게 작성하기
- Pull Request 전 코드 리뷰 & 테스트 수행하기


### 4.1 이름 규칙 (Naming Rules)

| 구분        | Python              | JavaScript & Java      | 기타         |
|-------------|---------------------|------------------------|--------------|
| PascalCase  | 클래스, Exception   | 클래스, Exception      | -            |
| camelCase   | -                   | 변수, 함수, 메소드     | -            |
| snake_case  | 변수, 함수, 메소드  | -                      | -            |
| kebab-case  | -                   | -                      | 파일, 폴더명 |
| UPPER_CASE  | 상수변수            | 상수변수               | -            |

- **Boolean 타입 변수 규칙**
    1. **상태** → `isRunning`, `isEnabled`
    2. **소유** → `hasData`, `hasChildNodes`
    3. **가능** → `canSubmit`, `canAccess`
    4. **권장/조건** → `shouldUpdate`, `shouldDisplay`


### 4.2 주석 및 커밋

#### 📝 주석 작성 원칙
- 불필요한 주석(코드 그대로 반복하는 설명)은 작성하지 않는다.
- 변경 시 주석도 **반드시 최신 상태**로 유지한다.

#### 💬 커밋 메시지 작성 원칙
- 의미 단위별로 작은 단위로 커밋한다.
- 커밋 메시지는 무엇을, 왜 변경했는지를 명확히 담는다.
- **Prefix 규칙**
    - `feat:` 새로운 기능 추가
    - `fix:` 버그 수정
    - `docs:` 문서 수정
    - `refactor:` 기능 변경 없는 코드 구조 개선
    - `test:` 테스트 코드 추가/수정
    - `style:` UI 컴포넌트 변경 


### 4.3 Pull Request

#### PR 생성 전 체크리스트
- 로컬에서 정상적으로 실행 확인
- 주요 기능 테스트 실행
- 불필요한 디버깅 코드와 주석 제거
- 코드 스타일 준수

#### PR 작성 원칙
- 변경된 주요 내용 요약 작성
- 필요 시 테스트 스크린샷/실행 로그 첨부
- Reviewer가 빠르게 이해할 수 있도록 간단한 설명 제공

#### 리뷰 시 유의사항
- 코드 품질 우선 확인
- 단순 비난이 아닌 개선 제안 중심 피드백
- Merge 전 최소 1명 이상의 리뷰 필요
